Quick Reference

A practical reference for training PyTorch vision models without running out of memory. Written for people who are new to GPU training — no systems background assumed. Use the Calculator tab for specific model and batch size estimates.

0 — Start Here

Don't read this whole document before you try anything. Apply these five changes first — they resolve the vast majority of GPU problems, and you can come back to understand the why behind each one.

  1. Turn on mixed-precision training.

    This is the single most impactful change you can make. It typically cuts memory by 35–45% and speeds training up 2–3× on any GPU made in the last five years. It takes about five extra lines of code and doesn't change your results.

    from torch.cuda.amp import autocast, GradScaler
scaler = GradScaler()

for images, labels in train_loader:
    optimizer.zero_grad()
    with autocast():                    # that's it for the forward pass
        loss = criterion(model(images), labels)
    scaler.scale(loss).backward()       # scaler handles the rest
    scaler.step(optimizer)
    scaler.update()
  2. Fix your DataLoader.

    By default, PyTorch loads data on a single thread. Your GPU finishes a batch and then sits idle while the CPU loads the next one. A few keyword arguments fix this.

    import os
DataLoader(
    dataset,
    batch_size         = 32,
    num_workers        = min(os.cpu_count() // 2, 4),  # was 0
    pin_memory         = True,
    prefetch_factor    = 2,
    persistent_workers = True,
    drop_last          = True,   # prevents a smaller final batch causing OOM
)
  3. Add torch.no_grad() to every validation loop.

    model.eval() alone is not enough. Without no_grad(), PyTorch still builds the full computation graph during validation — wasting memory you can't afford.

    model.eval()
with torch.no_grad():
    with autocast():            # still worth using during inference
        outputs = model(images)
  4. Use .item() when you log the loss.

    If you append the raw loss tensor to a list, you're keeping its entire computation graph in memory. Every epoch, the list grows.

    losses.append(loss.item())   # correct
losses.append(loss)          # wrong — graph stays alive
  5. Check memory before a long run.

    Two lines tell you how much GPU memory you're actually using vs. how much you've peaked at.

    print(round(torch.cuda.memory_allocated() / 1e9, 2), 'GB in use')
print(round(torch.cuda.max_memory_allocated() / 1e9, 2), 'GB peak')

    Use the Calculator tab to estimate whether a given model and batch size will fit before you start.

1 — What Is GPU Memory?

Your graphics card has its own dedicated memory — called VRAM — that's completely separate from your computer's RAM. Everything involved in a training step has to fit inside it at once. When it doesn't fit, you get a CUDA out of memory error and the run crashes.

Four things consume VRAM during training:

  • Model parameters. The weights of the network. Fixed in size once you've picked an architecture.
  • Activations. Intermediate results saved during the forward pass so the backward pass can compute gradients. Almost always the largest cost — it scales with batch size and image resolution.
  • Gradients. One gradient value per trainable parameter. Same size as the parameters. Zero for any frozen layer.
  • Optimizer state. Extra data the optimizer needs. Adam stores two extra tensors per parameter (running averages of past gradients). This can be surprisingly large.

Resolution is quadratic

If you double the image dimensions, activation memory doesn't double — it quadruples. A 512×512 image requires 16× more activation memory than a 128×128 one. This catches almost everyone off guard the first time.

Resolution doublesM_acts → 4×Quadratic. The most expensive thing to change.
Batch size halvesM_acts → ÷ 2Linear. Safe first response to an OOM error.

In practice: if you're hitting OOM, try halving the batch size first. If that doesn't help, try a smaller resolution. Only after both should you reach for more complex techniques.

2 — Mixed-Precision Training (AMP)

Modern GPUs have special hardware — Tensor Cores — that can do matrix multiplication in 16-bit floating point about 2–3× faster than in 32-bit. AMP takes advantage of this automatically, while keeping 32-bit precision for the parts of training where it matters most.

The result: activation memory is halved, most training steps run faster, and accuracy is essentially unchanged. See Section 0, Step 1 for the code.

What changesFP32AMP
Activation memoryBaseline~50% reduction
Total VRAMBaseline~55–65% of FP32
Max batch sizeBaseline~2× larger possible
Training throughputBaseline1.5–3× faster
AccuracyBaselineSame in practice

Why there's a GradScaler

FP16 can't represent very small numbers — anything below about 6×10⁻⁸ just becomes zero. Gradients in deep networks are often in this range, so without a workaround they'd silently vanish. The GradScaler multiplies the loss by a large number before backprop, keeping gradients in FP16's representable range, then divides back out before the optimizer step. It adjusts this scale factor automatically, so you never need to tune it.

If you see NaN losses with AMP, don't immediately blame the precision — check your learning rate first. AMP-induced NaNs from gradient underflow are less common than people expect, and are usually caught and skipped automatically by the scaler.

Optimizer memory

The optimizer is sometimes the second-biggest memory consumer. Adam stores two extra tensors per parameter (first and second moment estimates), both in FP32 regardless of whether you're using AMP. For a model with 300M parameters, that's about 2.4GB just for optimizer state.

# that's literally all you change

3 — Batch Size, Accumulation, and DataLoader

Gradient accumulation

When your desired batch size doesn't fit in memory, gradient accumulation lets you run several small batches before taking a single optimizer step. The GPU only ever holds one small batch at a time, but the model learns from the combined gradient of all of them.

Effective batchB_effective = B_micro × KK = steps between optimizer updates
Memory costcost of B_micro — not B_effectiveGPU never sees the full effective batch at once
Learning ratescale LR by (B_effective / original B)keep training dynamics equivalent
K = 8                          # pretend we have 8× the batch size
optimizer.zero_grad()

for step, (images, labels) in enumerate(loader):
    with autocast():
        loss = criterion(model(images), labels) / K   # divide — don't skip this
    scaler.scale(loss).backward()
    if (step + 1) % K == 0:
        scaler.step(optimizer)
        scaler.update()
        optimizer.zero_grad()
The division by K is not optional. Without it, the gradients accumulate to K times the correct magnitude and training usually diverges.

DataLoader settings

The GPU can't train faster than the CPU can load data. With num_workers=0 (the PyTorch default), the GPU finishes a batch and then sits idle while the CPU loads the next one. On a machine with a decent number of CPU cores, fixing this is often as impactful as any memory optimization.

SettingDefaultBetter valueWhat it does
num_workers02–4 per GPUParallel data loading — GPU stops waiting
pin_memoryFalseTrue~30% faster CPU-to-GPU transfer
prefetch_factor22–4Queues N batches per worker in advance
persistent_workersFalseTrueWorkers survive between epochs
drop_lastFalseTrue (training)Prevents smaller final batch causing OOM

CUDA prefetcher — hiding transfer latency

For maximum GPU utilization, a CUDA prefetcher sends the next batch to the GPU while the current batch is being processed, completely hiding the transfer time.

class CUDAPrefetcher:
    def __init__(self, loader):
        self.loader = iter(loader)
        self.stream = torch.cuda.Stream()   # separate stream for transfers
        self._preload()

    def _preload(self):
        try:   self.next = next(self.loader)
        except StopIteration: self.next = None; return
        with torch.cuda.stream(self.stream):     # transfer while GPU trains
            self.next = [t.cuda(non_blocking=True) for t in self.next]

    def __next__(self):
        torch.cuda.current_stream().wait_stream(self.stream)
        batch = self.next
        self._preload()          # kick off the next transfer immediately
        return batch

    def __iter__(self): return self

for images, labels in CUDAPrefetcher(train_loader):
    ...

Memory cost: one extra batch held on GPU. At B=32, 256×256, RGB with AMP, that's roughly 25MB — negligible.

4 — More Advanced Techniques

The things in this section are worth knowing about, but they're not step one. Get AMP and the DataLoader working first.

Gradient checkpointing

During the forward pass, PyTorch saves every intermediate activation so it can use them during backprop. Gradient checkpointing tells it to discard most of these and recompute them during the backward pass instead. You pay ~33% more in compute, but activation memory drops by 40–60%. Right tradeoff when you're memory-bound but not compute-bound.

Memory savingM_acts ÷ √L (approximate)L = number of layers. More layers = bigger benefit.
Compute costtraining time × ~1.33Worth it in most cases when memory is the bottleneck.
# HuggingFace models — one line
model.gradient_checkpointing_enable()

# Custom modules
from torch.utils.checkpoint import checkpoint
x = checkpoint(self.layer, x)   # activations recomputed in backward

Frozen backbone

When fine-tuning a pretrained model, you usually don't need to update every layer right away. Freezing the backbone means gradients aren't computed for those layers, so their activations don't need to be stored. This can halve the active memory cost.

for name, param in model.named_parameters():
    if 'classifier' not in name and 'head' not in name:
        param.requires_grad = False

# quick sanity check
trainable = sum(p.numel() for p in model.parameters() if p.requires_grad)
total     = sum(p.numel() for p in model.parameters())
print(f'Training {trainable/1e6:.1f}M of {total/1e6:.1f}M parameters')

A common schedule: freeze everything for a few epochs, then progressively unfreeze deeper layers. The backbone learning rate is usually 5–10× lower than the head.

PhaseTrainable layersMemoryWhen
Head onlyFinal layer(s) onlyMinimalEpochs 1–3
Top blocks + headHead + last 2–3 blocksModerateEpochs 4–6
Full modelEverything (low LR backbone)FullEpoch 7+

torch.compile() — free performance on PyTorch 2.0+

If you're on PyTorch 2.0 or later, this is an easy win. It compiles your model's computation graph ahead of time. The first forward pass is slower (compiling), but everything after is faster — sometimes significantly so.

model = torch.compile(model)
# first forward: slow. everything after: 20-100% faster.
# skip this during debugging — it makes tracebacks harder to read

5 — Classification vs. Segmentation

Classification — the straightforward case

Classification models output a single label per image. They're the most memory-efficient vision task because the model's output is tiny (one number per class) regardless of image size. Sections 0–4 are generally sufficient.

  • Resolution matters more than you'd expect. For natural image datasets, there's usually little accuracy difference between 224px and 256px — but the memory difference is real. Start at 224.
  • Frozen backbone fine-tuning works very well on smaller datasets. Full fine-tuning often overfits without aggressive augmentation.
  • For binary classification, the activation memory is identical to multi-class — only the final linear layer changes size. Going from 2 to 30 classes has almost no memory impact.

Segmentation — considerably more expensive

Segmentation models produce a class label for every pixel. The output tensor alone is batch × height × width × num_classes — for a 512×512 image with 20 classes, that's 20MB per image just for the output. U-Net architectures also hold onto all encoder feature maps for skip connections, which can triple or quadruple memory vs. a comparable classification model.

Patch-based training

For large images, train on random crops of a fixed size. Memory is then bounded by the crop size regardless of original image dimensions.

transforms.RandomCrop(512)    # memory is now always B × 512 × 512 × ...

# for inference on full images, stitch predictions with a sliding window:
def predict_full(model, image, patch=512, overlap=0.25):
    stride = int(patch * (1 - overlap))
    preds  = torch.zeros(1, num_classes, *image.shape[-2:]).cuda()
    counts = torch.zeros(1, 1,           *image.shape[-2:]).cuda()
    for y in range(0, image.shape[-2]-patch+1, stride):
        for x in range(0, image.shape[-1]-patch+1, stride):
            with torch.no_grad(), autocast():
                p = model(image[:,:,y:y+patch,x:x+patch])
            preds [:,:,y:y+patch,x:x+patch] += p
            counts[:,:,y:y+patch,x:x+patch] += 1
    return preds / counts

SAM and SAM2

SAM (Segment Anything Model) and SAM2 are foundation models for segmentation that operate at 1024×1024 resolution. A few things to know:

  • SAM ViT-H at batch=1 uses ~10–12GB with AMP — inference-only for most consumer GPUs. Fine-tuning requires freezing the image encoder and only training the prompt encoder and mask decoder (~4M of the 636M parameters).
  • SAM2 uses a Hiera backbone (hierarchical ViT that downsamples like a CNN), making it substantially more memory-efficient than SAM ViT at the same resolution. SAM2-Tiny at batch=1 is in the same range as SAM ViT-B.
  • Always use torch.no_grad() for SAM inference. It's easy to forget and the cost is enormous at 1024×1024.
The calculator includes presets for SAM ViT-B/L/H and all four SAM2 variants. Use it to check whether your GPU can handle inference at batch=1 before anything else.

6 — When Things Go Wrong

OOM errors are rarely mysterious once you've seen a few of them. There are really only about six things that cause them.

What happenedLikely causeFix
OOM on the first batchBatch size or resolution too largeEnable AMP first. If still OOM, halve the batch size.
OOM partway through epochVariable-size final batchAdd drop_last=True to DataLoader.
Memory climbs every epochStoring raw loss tensorsReplace append(loss) with append(loss.item()).
OOM during validation onlyMissing torch.no_grad()Wrap all validation code in with torch.no_grad():
OOM after resolution increaseActivation memory increased by (new/old)²Reduce batch size, or enable gradient checkpointing.
OOM with multiple GPUsDataParallel master GPU overheadSwitch to DistributedDataParallel.

The order to try things

When you're hitting OOM and need to reduce memory, apply these in order. Each step is safer and easier than the one after it.
